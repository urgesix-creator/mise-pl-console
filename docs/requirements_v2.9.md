> **【v2.9 改訂サマリー（v2.8 → v2.9）】**
>
> **販管費科目マスタの「上位分類」を、固定8区分からマスタ管理へ変更**した（実装済み・本番適用済み）。**経営データ・§8.1 税計算は不変**。
>
> 1. **上位分類は固定（人件費／賃料／…の8区分CHECK制約）ではなく、`expense_categories` マスタで管理**。運用で**自由に増減・改名・並び替え・無効化（=ソフト削除）**できる。改名はマスタ1箇所で全科目に反映。
> 2. **全社共通マスタ**（店舗非依存）。書込権限＝**経営層（executive）＋経理（accounting）**。読取は全ユーザー。
> 3. **使用中の区分は保護**（参照中は物理削除不可＝無効化で対応）。無効化しても既存科目・過去データは保持。
> 4. **科目側の上位分類は登録後変更不可**（区分の改名はマスタ側で全科目に反映）。店舗別追加科目・並び替え・無効化の既存仕様は不変。
> 5. データモデル設計書 **v1.11 §4.9・§4.9.1**、マイグレーション010（`expense_categories_master`）と整合。

> **【v2.8 改訂サマリー（v2.7 → v2.8）】**
>
> 2026-05-31 実装の「統合フォーマット使い勝手改善＋往復インポート」を設計書へ反映。**コード・DBは本更新では変更しない（設計書のみ）。§8.1 税計算ロジックの記述は不変。**

> 4. **往復インポートの実装**：プレビュー（ドライラン・書き込まない）→ 確認後に実書き込みの2段構成。**書き込み直前にサーバ側で再パース・再検証・再計算**（クライアント値を信用しない）。**空テンプレ行はドロップ**（エラー扱いしない）。
> 5. **権限の変更**：取込可能ロールは executive / country_rep / **accounting（経理も可）** / store_manager（staff 不可）。エクスポート権限と同条件に統一（旧記述の「経理不可」から変更）。
> 6. **境界判別は不変**：データ部ヘッダーは先頭セルが「店舗名」で始まる行で検出（変更なし）。

## 1. プロジェクト概要

### 1.1 目的

KOGAホールディングスの海外飲食店3店舗（タイ法人2店舗・ジャカルタ法人1店舗）の日次売上・仕入・販管費を、店長・現場社員が現場でWeb経由で入力し、本部がリアルタイムに把握できるシステムを構築する。既存のExcel運用を置き換え、現場入力の標準化・本部リアルタイム把握・月次PL自動化を実現する。

### 1.2 背景

- 現状はあお季タイ・博多天神ジャカルタそれぞれが独自フォーマットのExcelで運用されている
- 店舗ごとにフォーマット・仕入先・税制が異なり、本部での横断把握に時間がかかる
- 本部がリアルタイムに業績を把握できず、異常検知が遅れる
- KOGAグループ全体のAI活用方針（Claude2層構造設定の幹部展開等）と整合した、データ駆動型経営の基盤が必要

### 1.3 目的の優先順位

1. **現場入力の標準化・負荷軽減**（最優先）
2. 本部のリアルタイム業績把握
3. 月次PLの早期化・自動化
4. 全店横断のAI分析・パターン発見

## 4. 機能要件

### 4.1 認証・権限管理

- **認証方式**：メール＋パスワード（全ロール統一）
- 個別アカウント発行（共有ID禁止）
- パスワード要件：12文字以上、英数字記号混在
- 退職時：アカウント無効化処理（パスワード変更ではなく無効化）
- 重要操作（PL閲覧、マスタ変更等）時に2要素認証（2FA）併用
- ロールベースアクセス制御（RBAC）

【簡素化方針】アクセスログ・操作監査ログは記録しない（運用負荷軽減のため）。

### 4.2 日次入力機能

#### 4.2.1 売上入力（毎日必須）

**【v2.4：制度変更】** 売上は **ネットセールス（税抜売上）を主入力**とする。サービス料・税額は税抜売上から自動計算する。総売上（税込）は税抜と連動しない独立入力とする。

- **税抜売上（net_sales）**：**主入力値**（毎日必須）。予算比・概算粗利など業績指標の基準
- **サービス料（service_fee）**：**自動計算**（`税抜売上 × 店舗マスタのサービス料率`）
- **税額（tax_amount）**：**自動計算**（課税ベースは国マスタに従う。タイ＝税抜売上＋サービス料、インドネシア＝税抜売上＋サービス料。両国とも課税ベースは同一で、税率のみ異なる）
- **総売上（gross_sales・税込）**：**独立入力値**。税抜売上から自動算出しない（ネットと連動しない）
- 客数（customer_count）
- **客単価（avg_per_customer）**：**自動計算**（`総売上（税込）÷ 客数`。客数0の場合は表示なし）
- 昼夜区分（ランチ／ディナー）：店舗設定で切替可能（DB構造は維持）。**ただし v2.6 方針変更により、経営データは全店「1日1行（day_period='all'）」で入力する（下記注記）**
- 割引・インフルエンサー値引（任意項目、項目名カスタマイズ可）
- 天気・曜日・イベント（店舗設定で切替可能）
- 目標売上（税抜）との差額・達成率を画面表示

> **【v2.6 方針変更：経営データは全店「1日1行（day_period='all'）」に統一】**
> - 経営データ（`daily_sales`）は**全店で1日1行**（`day_period='all'`）で入力・保持する。**ジャカルタも今後は昼夜分離（lunch/dinner）では入力せず、1日合計で入力**する。
> - 時間帯（朝食・昼食・夕食等）や部門の細分は **「部門別売上（参考データ）」機能（§4.7）** で扱う。経営データ側では昼夜・部門を分けない。
> - `day_period` の `lunch`/`dinner` という**DB構造は残す**が、**今後は使わない**（構造削除はしない＝リスク回避のため）。
> - これに伴い、従来「day_period は経営 net の入力粒度」とした位置づけは**「全店 all 単位」に変更**（データモデル設計書 v1.9 §4.10 の補足と整合）。

> **【入力値・自動計算値の整理（v1.7 §8.1 と整合）】**
> - **主入力**：税抜売上（net_sales）
> - **独立入力**：総売上（税込）、昼夜分離店のランチ売上（税込）／ディナー売上（税込）。いずれも税抜と一致・連動を要しない
> - **自動計算**：サービス料（＝税抜×料率）、税額（課税ベースに従う）、客単価（＝税込÷客数）
> - **整合性**：税抜と税込の厳密一致は要求せず、桁違いのみ警告（§4.5.6 参照）

#### 4.2.2 仕入入力（仕入発生日のみ任意）

- 仕入先別明細（店舗別マスタから選択）
- 仕入カテゴリ別合計（自動集計）
- 当日・累計の自動計算

#### 4.2.3 販管費入力（毎日入力）

- 給与・家賃・光熱費・減価償却・広告宣伝・車リース・会計・警備等
- 店舗別マスタで科目カスタマイズ可能

- **管理単位**：店舗ごとに総額1金額（将来的にカテゴリ別へ拡張可能な構造）
- **入力場所**：日次入力画面の仕入欄の横に並列配置
- **表示形式**：「現在の概算棚卸：◯◯◯,◯◯◯ {通貨}（◯月◯日更新）」を常時表示
- **更新方法**：編集ボタンで金額を上書き更新。日々の入力は不要
- **値の保持**：店舗ごとに**最新の現在値1件のみ**（履歴は保持しない、更新時は上書き）
- **PL計算への適用**：MVPでは月次PL売上原価計算に**使用しない**（売上原価＝当月仕入合計の簡略計算）。在庫資産把握用の参考値として表示
- **権限**：店長以上が更新可能。現場社員は閲覧のみ
- **リマインダー（必須運用）**：月末日にSlack経営会議グループへ「概算棚卸の更新確認」を自動通知。月末更新がない場合、翌月1日に未更新アラートを再通知
- **強制運用設定**：上記リマインダーは初期設定で「必須」として動作。管理者権限ユーザーがシステム設定画面から「任意（オプション）」へ後日変更可能

【簡素化方針】変更履歴は記録しない。最終更新日時のみ自動記録。

#### 4.5.1 日報・週報（優先度1）

**日報（毎朝自動配信）**

- 配信先：Slack経営会議グループ

- 報告内容：店舗ごとに「月初〜前日までの日別累計」を報告

**日報フォーマット仕様：**

```
【日報】YYYY年M月D日（曜日）

▼ {店舗名}（{通貨コード}）月次予算 {通貨記号}{月次予算額}
{日付}({曜日}) 売上{当日売上}／累計{累計売上}／予算比{予算比}％／粗利{当日粗利}／累計粗利{累計粗利}
{日付}({曜日}) 売上{当日売上}／累計{累計売上}／予算比{予算比}％／粗利{当日粗利}／累計粗利{累計粗利}
...（前日まで日別に出力）

▼ {次の店舗名}（同様の形式）
...
```

**指標定義：**

| 指標 | 計算式 | 備考 |
|---|---|---|
| 当日売上 | 該当日の税抜売上（net_sales） | 業績把握の標準指標 |
| 累計売上 | 月初〜該当日までの税抜売上合計 | |
| 累計予算 | SUM(daily_targets.target_sales) WHERE 月初 ≤ target_date ≤ 前日 | 日別予算の合計（曜日変動を正確に反映）。目標も税抜基準 |
| 予算比 | 累計売上 ÷ 累計予算 × 100 | 整数％表示。売上・予算とも税抜基準で同一 |
| 当日粗利（概算） | 該当日の税抜売上 − 該当日の仕入合計 | 概算値。月末棚卸更新時に月次粗利は別途調整 |
| 累計粗利（概算） | 月初〜該当日までの概算粗利合計 | 同上 |

【注意】日報の粗利は「概算値」。月次PLの正式な売上総利益（売上原価＝月初棚卸＋月間仕入−月末棚卸を使用）とは異なる旨を、日報フッターに毎回明記する。

【v2.4 注記】日報の売上・予算比は v2.3 から変更なし（従来より税抜売上＝net_sales 基準で一貫）。今回の制度変更（税抜を主入力に確定）により、入力値と日報指標の基準が完全に一致した。

**フォーマットの設計原則：**

- プレーンテキスト形式（Slack表示で崩れないため、テーブル形式は使用しない）
- 1日1行で簡潔に表示
- 通貨記号は店舗の通貨マスタから自動取得
- 数値は3桁カンマ区切り
- 店舗順は管理者設定で指定可能（デフォルトは店舗マスタの登録順）

**週報（毎週月曜朝に自動配信）**

##### 4.5.5.1 日次売上 往復フォーマット（v2.7 確定＝実装済みの「統合フォーマット」）

日次売上（`daily_sales`）と部門別売上（`daily_department_sales`）を **1枚のExcelに横統合**した「統合フォーマット」を、往復（エクスポート⇄インポート）の正典とする。エクスポートは実装済み、インポート本体は別タスク（同一レイアウトで往復互換に設計済み）。

> **【v2.7 注記】** v2.6 では「日次売上のみ・平坦な単一表」だったが、実装は **経営データ＋部門別売上の統合表（店舗ごと1ファイル・部門横展開・合計行・セル色分け）** に発展した。本節はその**実装の実態**に合わせて確定したものである。**DBスキーマ（テーブル列・制約）の変更は伴わない。**

**実装・スコープ方針：**

- **実装形態**：アプリ内機能（出力ボタン）。データ画面（/data）の「日次売上エクスポート（統合）」セクションに配置。CLIスクリプトではない。
- **形式**：**Excel（.xlsx）のみ**。色付け（セル背景）のため **ExcelJS** を使用（部門別の集計CSV/ExcelエクスポートはSheetJSのまま別系統で温存）。
- **対象**：**日次売上（daily_sales）＋部門別売上（daily_department_sales）** を1枚に統合（仕入・販管費は将来の横展開）。
- **単位**：**店舗ごと（1店舗＝1ファイル）**。部門は店舗固有のため、統合版は店舗必須（「全店一括」は廃止）。
- **通貨・税率**：**`store_id` から店舗を特定し、システムが自動処理**（タイ=THB/VAT7%、ジャカルタ=IDR/PB1 10%）。Excel側に通貨・税率・JPY換算列は持たない（**ローカル通貨のみ**）。
- **店舗識別**：人が読めるよう **店舗名を表示**しつつ、照合用に **`store_id` を併記**。取込時は **`store_id` で照合**する。

**シート構成：**

1. **見出しブロック（先頭）** — 以下を各行に置く。
   - タイトル「**日次売上（統合）**」
   - **対象期間**：`対象期間：YYYY-MM-DD 〜 YYYY-MM-DD`
   - **対象店舗**：`対象店舗：{店舗名}（store_id: {uuid}）` — **store_id をここに1回だけ表示**（データ行には持たない）。取込はこの値で店舗を照合する。
   - **日付ガイド**：`日付は YYYY-MM-DD 形式（文字列）で入力してください。`
   - **凡例**：`【必須】未記入不可（グレー） ／ 【任意】入れれば反映（色なし） ／ 【自動】入力不要・自動計算（薄いブルー）`
   - 注記：`このファイルはインポート（取込）にも使えます。`
   - **天気の選択肢**：`天気の選択肢：晴 / 曇 / 雨 / 雪 / その他`（記入者がコピペで使う。売上入力画面の定義と同期）
   - 空行（見出しとデータ部の区切り）
2. **データ部** — ヘッダー行（列見出し）＋日付ごと1行＋末尾に合計行。
   - **全日付テンプレート出力**（v2.8）：エクスポートは対象期間 from〜to の**全日付**を行として出力する（記入用テンプレート）。データのある日は値入り、無い日は**日付・店舗名のみの空行**。**データが0件の期間でも全日付テンプレートを出力する**（空ファイルにはしない。期間不正 from>to のみエラー）。

**データ列定義（経営データ・実DBカラム基準）：**

| 列（見出しタグ） | 対応カラム | 区分(色) | 入力(取込) | 出力 | 必須 | 説明 |
|---|---|---|---|---|---|---|
| 店舗名【必須】 | （stores.name） | 必須(グレー) | 参照（表示用） | ○ | － | 人が読むための表示（**データ行に残置**）。照合には使わない |
| 日付【必須】 | business_date | 必須(グレー) | キー | ○ | ○ | YYYY-MM-DD。**文字列（numFmt='@'）で出力**し、Excel の自動日付変換を抑止 |
| 昼夜区分 | day_period | 任意(色なし) | キー | ○ | － | **全店 `all` 前提**（lunch/dinner は今後不使用） |
| 税抜売上【必須】 | net_sales | 必須(グレー) | **主入力** | ○ | **○** | 新制度の主値（予算比・粗利の基準） |
| 税込売上【任意】 | gross_sales | 任意(色なし) | 独立入力 | ○ | △ | 任意。空欄なら 0。`net_sales` と非連動（逆算しない） |
| 客数【任意】 | customer_count | 任意(色なし) | 入力 | ○ | △ | 0以上の整数 |
| 天気【任意】 | weather | 任意(色なし) | 入力 | ○ | × | **日本語ラベル（晴/曇/雨/雪/その他）で表示**。DBは英字コード保存・出力時に変換 |
| イベントメモ【任意】 | event_note | 任意(色なし) | 入力 | ○ | × | 自由テキスト（最大500字）。**入力画面の「イベントメモ」と同一フィールド**。取込は「イベントメモ」「備考」両ラベルを受理（後方互換） |
| サービス料【自動】 | service_fee | 自動(薄ブルー) | **取込しない** | ○（参考） | － | 自動計算値。取込時は無視しサーバ側で再計算 |
| 税額【自動】 | tax_amount | 自動(薄ブルー) | **取込しない** | ○（参考） | － | 自動計算値。取込時は無視しサーバ側で再計算 |
| 客単価【自動】 | （算出値） | 自動(薄ブルー) | **取込しない** | ○（参考） | － | `gross_sales ÷ customer_count`（参考表示・DB列ではない） |

- **store_id はデータ行に持たない**（v2.8 改訂）：1店舗1ファイルのため、store_id は**見出しブロックに1回だけ表示**する。取込はその見出しの store_id で店舗を照合する（データ行に `store_id` 列は無い）。**店舗名列は表示用にデータ行へ残置**する。

**データ列定義（部門別・横展開）：**

| 列（見出しタグ） | 対応 | 区分(色) | 入力(取込) | 出力 | 説明 |
|---|---|---|---|---|---|
| {部門名}【任意】 …（有効部門ぶん） | daily_department_sales.gross_sales | 任意(色なし) | 入力 | ○ | その店の**有効部門（is_active=true）を display_order 順**に横展開。各部門の税込売上 |
| 部門計【自動】 | （算出値） | 自動(薄ブルー) | **取込しない** | ○（参考） | その日の部門別の合計（参考表示） |

- **経営データと部門別は連動しない**。各データの実値をそのまま表示し、`部門計` が `gross_sales` と一致する必要はない。無効化部門は列に出さない。

**合計行（末尾）：**

- 数値列（税抜・税込・客数・サービス料・税額・各部門・部門計）は**期間内の単純合計**。
- **客単価の合計は足し込まず**、`税込合計 ÷ 客数合計` で**再計算**する（平均の合計は無意味なため）。

**セル色分け（ExcelJS）：**

- **グレー `FFE5E7EB`＝必須**：店舗名 / 日付 / 税抜売上(net)（**store_id は見出しへ移動したため列にはない**）
- **薄ブルー `FFDBEAFE`＝自動計算・反映されない**：サービス料 / 税額 / 客単価 / 部門計
- **色なし＝任意**：税込売上 / 客数 / 天気 / イベントメモ / 各部門
- 色非対応ビューア対策として、**列見出しに文字タグ【必須】【任意】【自動】を併記**する。

**データ部の境界判別（往復インポート用）：**

- インポートは、各行の**先頭セルが「店舗名」で始まる行**を「データのヘッダー行」として検出し、それより上（見出しブロック）を**読み飛ばす**。見出しの文言・行数が変わってもこの方式で開始行を再発見できる。

**取込（インポート）方針（v2.6 から不変・再掲）：**

- **net 主入力**：取込の主値は `net_sales`（必須）。`gross_sales` は任意（空欄なら 0、`net_sales` とは独立）。`service_fee`/`tax_amount`/客単価は**取込時は無視し、`net_sales`・店舗のサービス料率・国の税率／課税ベースからサーバ側で再計算**（データモデル設計書 §8.1 の新制度に従う）。
- **マッチング**：`(store_id, business_date, day_period)`・**常に上書き（UPSERT）**・ファイルに無い行は変更しない。部門別は `daily_department_sales`（`(store_id, business_date, department_id)`）へUPSERT。
- **day_period**：本フォーマットは**全店「1日1行（`day_period='all'`）」前提**（§4.2.1 の全店all統一方針）。昼夜分離（lunch/dinner）では入力しない。
- **天気**：日本語ラベル⇄英字コードの相互変換は売上入力画面と同一定義を参照。**選択肢は 晴/曇/雨/雪/その他（「大雨」は採用しない）**。規定外ラベルは**原文を温存しつつ警告のみ**（取込はブロックしない）。
- **空テンプレ行のドロップ**（v2.8）：日付（・店舗名）だけで売上系（net/gross/客数/天気/イベントメモ/部門）が全て空の行は「未記入の日」として**取込対象から除外（ドロップ）し、エラーにしない**。一方、**一部でも記入があり `net_sales` が欠落している行はエラー**（その行のみスキップ）。
- **ファイルレベルの店舗照合**（v2.8）：見出しの store_id が**読み取れない、または取込先として選んだ店舗と一致しない**場合は、ファイル全体をエラーにする（誤ファイル防止）。
- **桁違い警告**：`gross_sales ÷ net_sales` が通常範囲外なら警告のみ（保存はブロックしない・§8.1.1）。

**権限：**

- エクスポートは**現場社員（staff）不可・店長は自店のみ・経営層（および country_rep / accounting）は全店**（§4.5.5 共通要件に準拠）。判定はサーバ側で実施し、RLS（`can_access_store`）が行範囲を多重防御で担保する。

#### 4.5.6 Excel一括取込（インポート）機能

ダウンロードしたExcel形式のデータをテンプレートとして、一括登録・修正を可能とする。Excel出力機能とは双方向の往復互換を持つ。

> **【v2.6】日次売上の取込は §4.5.5.1 の往復フォーマットに従う。** 主値は `net_sales`（税抜・必須）、`gross_sales` は任意（空欄なら0・net と非連動）。`service_fee`/`tax_amount` は取込時は無視し**サーバ側で再計算**（§8.1）。店舗は **`store_id` で照合**。経営データは**全店「1日1行（`day_period='all'`）」前提**（§4.2.1）。

> **【v2.8】日次売上 統合フォーマットの往復インポートを実装済み。** 入力は §4.5.5.1 の統合フォーマット。確定仕様は以下（**DBスキーマ変更なし・§8.1 税計算は不変**）。
>
> - **2段構成（安全策）**：① プレビュー（ドライラン）＝サーバ側でパース・検証・再計算して**書き込まずに**結果を返す（新規/上書き/エラー/スキップ件数・上書き差分・再計算値・未登録スキップ部門を提示）。② 実書き込み（確認後）＝プレビューと同一ファイルを送信し、**書き込み直前にサーバ側で再パース・再検証・再計算**（**クライアントの計算値は信用しない**）。UI は**確認ダイアログ必須**（**上書き件数を強調・元に戻せない旨を警告**）。
> - **書き込み先**：`daily_sales`（経営）と `daily_department_sales`（部門別）の**2テーブルのみ**。マッチングは `daily_sales`:`(store_id, business_date, day_period='all')`／`daily_department_sales`:`(store_id, business_date, department_id)`。**常に上書き（UPSERT）・ファイルに無い行は変更しない（DELETE しない）**。
> - **保存値**：`net_sales`（主入力）・`gross_sales`（独立入力・空欄0）・`customer_count`・`weather`・`event_note` を取込。**`service_fee`/`tax_amount` は §8.1 の再計算値を保存**。**客単価（avg_per_customer）は `daily_sales` に列が無いため保存せず、表示のみ**。
> - **行の扱い**：エラー行はスキップ（正常行のみ書込）／未登録部門列はスキップ（自動登録しない）／**空テンプレ行はドロップ**／ファイル内キー重複は最後の行優先。
> - **イベントメモ列の用語統一**：エクスポートの列見出しは「**イベントメモ**」（入力画面の「イベントメモ」＝`event_note` と一致）。**取込は「イベントメモ」「備考」の両ラベルを `event_note` として受理**（旧フォーマット後方互換）。データ構造・保存・§8.1 は不変。
> - **トランザクション（2テーブル非原子）**：テーブル単位のバルクUPSERT（`daily_sales` を先、`daily_department_sales` を後）。**2テーブルを跨ぐ厳密原子化は行わない**。いずれも冪等のため**途中失敗時は同じファイルで再取込すれば回復**。失敗時はどこまで反映されたかを明示する。
> - **権限（v2.8）**：取込可能は **executive / country_rep / accounting（経理も可）/ store_manager**。**staff 不可**（UI 非表示＋サーバ拒否）。country_rep は自国、store_manager は自店のみ。RLS が多重防御。**※旧記述の「取込は経理不可」から、v2.8 で経理も可に変更（エクスポート権限と統一）**。

**取込種別：**

| # | 取込種別 | 対象データ | 出力との対応 |
|---|---|---|---|
| 1 | 日次データ取込 | 売上・仕入・販管費（指定期間・指定店舗） | 4.5.5 #1 出力と互換 |
| 2 | **日別売上目標取込** | **指定月・指定店舗の日別売上目標** | **4.5.5 #6 出力と互換** |

**用途：**

| # | 用途 | 想定シーン |
|---|---|---|
| 1 | 既存Excelからの初期データ移行 | 既存「あお季タイ」「博多天神ジャカルタ」のExcelデータを新システムへ一括移行 |
| 2 | 過去データの後追い入力 | システム導入後、過去分のデータを遡って一括登録 |
| 3 | 一括修正・編集（何度でも実行可能） | 出力→Excelで編集→再取込するラウンドトリップ運用 |
| 4 | 年間予算の一括設定 | 12ヶ月分の日別売上目標をExcelで設計→取込 |
| 5 | 前年実績ベースの予算策定 | 前年の日次データを出力→翌年予算欄を編集→目標として取込 |

**処理方針：常に上書き（OVERWRITE）に統一**

- マッチするレコードは**常に上書き**（旧：上書き/スキップ選択方式は廃止）
- 何度でも修正取込可能：**最新の取込が常に正となる**
- 取込ファイルに含まれないレコードは**変更されない**（部分更新OK）

**マッチング条件（テーブル別）：**

| テーブル | マッチング条件 | 動作 |
|---|---|---|
| daily_sales | (store_id, business_date, day_period) | 一致レコードを上書き、無ければ新規作成 |
| daily_purchases | (store_id, business_date, supplier_id) | 同上 |
| daily_expenses | (store_id, business_date, expense_account_id) | 同上 |
| daily_targets | (store_id, target_date) | 同上 |

**処理フロー：**

1. ファイルアップロード（ドラッグ＆ドロップ または ファイル選択）
2. 取込種別の自動判定（または手動選択）
3. バリデーション実行
4. プレビュー画面表示（影響を受けるレコード数：新規作成X件、上書きY件、エラーZ件）
5. 確認ボタン押下で取込実行（UPSERT）
6. 結果レポート表示

**バリデーション項目：**

- ファイル形式（.xlsx）
- 必須カラムの存在
- データ型（日付・数値）
- 値の妥当性（負の売上等はエラー）
- 仕入先・カテゴリのマスタ照合
- マッチング条件の重複（同一ファイル内で同じキーが複数あれば最後の行が優先）
- **売上の桁違い警告（v2.4：制度変更）**：総売上（税込）÷ 税抜売上 が **2.0 超** または **0.5 未満**の場合に「桁違いの可能性」を**警告表示**する。ただし**取込・保存はブロックしない**（税抜と税込は独立入力のため、厳密一致は検証しない）。税抜売上が 0 または未入力の行は判定しない

> **【v2.4 注記】** 旧 v2.3 の「計算整合性チェック（税抜＋税＝税込、合計値の矛盾検出）」は**廃止**。総売上（税込）は税抜から自動算出しない独立入力値となったため、厳密な等式チェックは行わず、入力ミス検出を目的とした桁違い警告のみとする（データモデル設計書 v1.7 §8.1.1 と整合）。

**マスタ未登録項目の処理：**

- 取込データに未登録の仕入先名・カテゴリ名がある場合、プレビュー時にマッピング画面表示
- 選択肢：「既存マスタにマップする／新規マスタとして自動登録する／その行をスキップする」
- 一括処理オプションあり

**権限制御：**

| ロール | インポート権限 |
|---|---|
| 経営層 | 全店インポート可（日次データ・売上目標とも） |
| 各国法人代表 | 自国店舗のみインポート可 |
| 店舗店長 | 自店のみインポート可 |
| 現場社員 | インポート不可 |
| 経理・税理士 | インポート不可（閲覧・出力のみ） |

**監査・安全機能（簡素化）：**

- インポート実行前のプレビュー画面で確認（**プレビューが唯一の安全機構**）
- プレビュー画面で「すべて上書き」「すべてスキップ」等の一括操作を慎重に確認すること
- **【重要】取込実行後は元に戻せない**ため、プレビュー段階での確認が極めて重要

【簡素化方針】取込履歴・スナップショット・24時間ロールバック機能は実装しない。

### 4.7 部門別売上（参考データ）機能（v2.5追加）

**経営計算から完全に独立した参考機能**。損益・原価率・予算比・客単価には一切流入しない。

**目的**
- 部門別（朝食・昼食・夕食・デリバリー等）の売上を把握し、**人員配置・メニュー構成の判断の参考**にする。

**入力**
- 入力単位：**店舗 × 営業日 × 部門**。値は**税込売上のみ**（部門ごと）。
- **任意入力**（空欄可）。部門別を入力しなくても経営売上（日次入力）の保存は成立し、**保存をブロックしない**。
- 同日同部門の再入力は**UPSERT 上書き**（加算しない）。
- 通貨は**店舗ローカル通貨（税込）のみ**。JPY換算は行わない。
- 入力UIは日次売上入力画面に「部門別売上（参考）」セクションとして組み込む（画面設計書 v1.2 §参照）。

**部門マスタ**
- **店舗ごとに完全独立**。共通初期値なし。各店が運用開始時にゼロから登録（追加・改名・並べ替え・無効化）。
- 削除は**ソフト削除（無効化）**。過去データは保持し、集計・構成比に引き続き含める。
- マスタ画面に**登録例**（朝食／昼食／夕食／デリバリー等）をガイド表示。

**構成比（部門内シェア）**
- **構成比 = 各部門の累計税込売上 ÷ 部門別累計合計**（同一店舗・同一期間）。**全部門の合計は100%**。
- **累計ベース**。期間既定は「当月初〜直近」、UIで手動変更可。
- 無効化済み部門でも、過去データを持つ場合は集計・構成比に**含める**。
- 注意：分母は「部門別売上の累計合計」であり、`daily_sales` の1日合計ではない（経営データと独立）。

**整合性**
- 部門別売上の合計が1日合計（`daily_sales`）と**一致する必要はない**（検算・整合性チェックなし）。

**権限**
- 部門別売上の**入力**：日次売上を入力できる人と同一（現場社員も可）。
- 部門マスタの**編集**：店長以上（executive / country_rep / store_manager）。
- **閲覧**：自店のみ／経営層は全店（既存の店舗別データと同じ範囲）。

**抽出・表示**
- 既存「データ」画面に統合して表示（部門×日のクロス集計・期間累計・構成比・CSV/Excelエクスポート）。詳細は画面設計書 v1.2。

## 5. データモデル設計方針（4レイヤー構造）

### 5.1 レイヤー1：全店共通の必須コア

- 店舗ID・日付・通貨・基準為替レート
- **売上：税抜売上（net_sales・主入力）/サービス料（自動計算）/税額（自動計算）/総売上（税込・独立入力）**
- 客数・客単価（客単価＝総売上（税込）÷ 客数）
- 仕入合計・消耗品合計・販管費合計
- **概算棚卸金額（店舗ごとに最新値1件のみを保持。履歴は持たない）**

> **【v2.4 注記】** 売上の主入力は税抜売上（net_sales）。総売上（税込）は税抜と連動しない独立入力。サービス料・税額は税抜から自動計算（詳細は §4.2.1 およびデータモデル設計書 v1.7 §8.1）。

### 5.2 レイヤー2：店舗設定で切替可能なオプション

- 昼夜区分（ON/OFF）：タイは現在OFF、ジャカルタはON、将来切替可能。ON時はランチ・ディナーを税込で独立入力（day_period 行方式で保持）
- 天気入力（ON/OFF）：タイは現在ON、ジャカルタはOFF
- イベント入力（ON/OFF）：ジャカルタON
- 割引・インフルエンサー値引（項目名カスタマイズ可）

### 5.3 レイヤー3：店舗別マスタ（完全カスタマイズ）

- **店舗マスタ：店舗ごとのパラメータレコード（通貨、サービス料率、税制適用国、昼夜区分ON/OFF、天気入力ON/OFF、イベント入力ON/OFF）**
- 仕入先マスタ：店舗ごと完全独立
- 仕入カテゴリマスタ：店舗ごと自由定義
- 販管費科目マスタ：店舗ごと自由定義
- 目標マスタ：店舗ごと月次設定

### 5.4 レイヤー4：国マスタ（税制対応）

| 国 | 税名称 | 税率 | 課税ベース |
|---|---|---|---|
| タイ | VAT | 7% | 税抜売上＋サービス料 |
| インドネシア | PB1（JKT地方税） | 10% | 税抜売上＋サービス料 |

> **【v2.4 注記】** 税額は課税ベースを基準に自動計算する。タイ・インドネシアとも `(税抜売上 + サービス料) × 税率`（課税ベースは同一、税率のみタイ7%・インドネシア10%で異なる）。サービス料は `税抜売上 × サービス料率`（ネット売上に課す）。データモデル設計書 v1.7 §8.1 と整合。課税ベースの内部値はタイ・インドネシアとも `net_plus_service`（`net_sales` は日本・台湾等の将来用）。

- 通信障害時は入力不可（運用ガイドで明示）
- クラウドサービスの稼働率に準拠（99.9%程度）

### 6.4 多通貨対応

- 入力通貨：THB（タイ）／IDR（インドネシア）
- 表示通貨：原通貨＋JPY換算
- レート：管理者が手動入力（月末等に更新）

#### 為替レート運用ルール（重要）

**「月末レート」として運用：1ヶ月1レート、当月の全データに統一適用**

| 観点 | ルール |
|---|---|
| 概念モデル | 当月内の全データは、登録されている**最新のレート1値**で換算 |
| 月途中の更新 | 月途中でレートを更新した場合、**当月の過去データも遡って新レートで再計算** |
| 履歴管理 | 過去レートの履歴は保持しない（簡素化方針） |
| 月跨ぎ | 翌月になっても自動切替なし。実務上は月末に更新する運用を推奨 |
| 適用箇所 | 月次PL生成・ダッシュボードのJPY換算表示・データ閲覧のJPY参考表示 |

**運用例：**

```
5月1日 月初  ：1 THB = 4.70 円 で登録
5月15日      ：当月の全売上が4.70円で換算表示される
5月31日 月末 ：1 THB = 5.00 円 に更新
            → 5月1日〜31日の全データが遡って5.00円換算となる
            → 月次PLは5.00円で集計
```

**運用上の判断ポイント：**

- 月内のJPY換算表示は「暫定値」として位置付け
- 月末確定のJPY換算は「月末レート更新後」が正
- 簡略化のため、月内の細かいレート変動は反映しない

## 11. 技術スタック

### 11.1 採用方針

比嘉俊一がClaude Codeで単独開発・運用する前提で、**マネージドサービス中心・Claude Code適合性最大化**の方針で技術選定。

|---|---|---|
| フレームワーク | Next.js（App Router） | 14+ |
| 言語 | TypeScript | 5+ |
| UIライブラリ | shadcn/ui + Tailwind CSS | 最新安定版 |
| フォーム管理 | React Hook Form + Zod | 最新安定版 |

#### データ・認証・ストレージ

| 層 | 技術 | 備考 |
|---|---|---|
| データベース | Supabase（PostgreSQL） | 認証・ストレージ統合 |
| 認証 | Supabase Auth | メール+パスワード、2FA対応 |
| 権限制御 | Row Level Security（RLS） | DB層で権限強制 |
| ファイル保存 | Supabase Storage | Excel取込ファイル等 |

#### 機能別パッケージ

| 用途 | パッケージ |
|---|---|
| Excel入出力 | ExcelJS |
| Slack連携 | @slack/web-api |
| 日付・タイムゾーン | date-fns + date-fns-tz |
| 金額計算（誤差防止） | decimal.js |

#### ホスティング

| 用途 | サービス | 想定コスト |
|---|---|---|
| アプリ | Vercel | 無料〜$20/月 |
| データベース | Supabase | 無料〜$25/月 |
| ドメイン | お名前.com等 | $10/年 |

### 11.3 採用根拠

1. **Claude Code適合性が極めて高い**：Claude Codeが最も訓練されているスタック。単独開発時の開発速度最大化
2. **マネージドサービス中心**：サーバー管理・OS更新・バックアップ等インフラ管理ほぼゼロ
3. **初期コスト極小**：実質月$0で開始可能、データ拡大しても月$45程度
4. **行レベルセキュリティ（RLS）でDB層から権限強制**：アプリ層のバグでも財務データが漏洩しない
5. **拡張性・将来性**：世界で最も採用されているスタックの1つ。Phase3でAI機能追加も容易

### 11.4 検討した代替案と不採用理由

| スタック | 不採用理由 |
|---|---|
| AWS Amplify | インフラ複雑、単独運用には過剰 |
| Firebase | NoSQL（Firestore）が複雑な集計に不向き |
| Convex | リアルタイム機能不要、エコシステム小 |
| PocketBase（自己ホスト） | 自己ホスティングで保守負担増 |
| Rails / Django | フルスタックフレームワークは単独開発で過剰 |
