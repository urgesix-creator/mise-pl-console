要件定義書v2.4で確定したMVP範囲（13機能）に対応する**画面アーキテクチャ**を規定する。本書はフェーズ5aの成果物であり、画面の数・役割・関係性・ナビゲーションを定義する。

### 6.3 日次入力画面（#5）—— **最重要画面**

| 項目 | 内容 |
|---|---|
| 想定利用者 | 店舗店長・日本人マネージャー・現場社員 |
| 主要機能 | 売上・仕入・販管費・概算棚卸を1画面で入力 |
| レイアウト | 縦スクロール式（タブ式は採用しない、モバイルで操作しづらいため） |
| 構成セクション | ①基本情報 ②売上 ③仕入 ④販管費 ⑤概算棚卸 ⑦部門別売上（参考・v1.2追加） ⑥保存 |

**詳細セクション仕様：**

#### ①基本情報セクション
- 日付選択（デフォルト：今日。過去日も選択可）
- 店舗名（複数店舗権限保有者のみ切替可、店長は固定表示）
- 天気選択（店舗設定でON/OFF）
- イベントメモ（店舗設定でON/OFF）

#### ②売上セクション

**【v1.1：制度変更】** 売上は**税抜売上（ネットセールス）を主入力**とする。総売上（税込）は税抜と連動しない独立入力。サービス料・税額・客単価は自動計算して表示する。

- **税抜売上（net_sales）入力欄** ← **主入力**（必須）。予算比・粗利の基準
- **総売上（税込・gross_sales）入力欄** ← **独立入力**。税抜から自動算出しない（ネットと連動しない）
- 客数入力欄
- 昼夜区分があるケース（ジャカルタ）：**ランチ売上（税込）／ランチ客数／ディナー売上（税込）／ディナー客数を分離入力**。ランチ売上・ディナー売上はいずれも**税込で独立入力**し、税抜（net）とは連動しない（day_period 行方式で保持）
- **自動計算表示：サービス料・税額・客単価**
  - サービス料 ＝ 税抜売上 × 店舗マスタのサービス料率
  - 税額 ＝ タイ・インドネシアとも：（税抜売上＋サービス料）× 税率（課税ベースは同一、税率のみタイ7%・インドネシア10%）
  - 客単価 ＝ **総売上（税込）÷ 客数**（客数0の場合は表示なし）
- 割引・インフルエンサー値引（任意項目）
- 当日の売上目標（税抜）と達成率を表示

> **【入力欄／自動計算表示の区別（v1.7 §8.1・v2.4 §4.2.1 と整合）】**
> - **入力欄（手入力）**：税抜売上（主入力）、総売上（税込・独立）、客数、（昼夜分離店）ランチ売上・ディナー売上（いずれも税込・独立）と各客数
> - **自動計算表示（読取専用）**：サービス料、税額、客単価
> - 旧 v1.0 では「総売上（税込）」が主入力欄、「税抜売上」が自動計算表示だったが、本改訂で主従を逆転。税抜が入力、税込は独立入力に変更。

#### ③仕入セクション
- 「仕入を追加」ボタン
- 1件ごとに：仕入先（プルダウン、店舗別マスタから選択）＋金額
- 「仕入なし」チェックボックス（仕入なしの日はチェック1つで完了）

#### ④販管費セクション
- 店舗の販管費科目マスタを基にした科目別入力
- 科目ごとに金額入力欄（0円の科目は省略表示）
- 「展開／折りたたみ」で全科目表示切替

#### ⑤概算棚卸セクション
- 現在値表示「現在の概算棚卸：XXX,XXX THB（5月1日更新）」
- 「更新する」ボタン → モーダルで新しい金額入力
- 履歴表示リンク（直近変更を一覧表示）

#### ⑥保存
- 「保存」ボタン（明示的保存、自動保存はしない）
- バリデーションエラーは保存ボタン押下時に一覧表示
- 保存成功後は確認トースト表示
- 「前日の値をコピー」ボタン（入力負荷軽減UI）

| 設計上の重要点 | 内容 |
|---|---|
| バリデーション | 必須項目チェック（税抜売上・客数）、数値範囲（負値はエラー）、**売上の桁違い警告のみ**（下記） |
| 桁違い警告（v1.1） | 総売上（税込）÷ 税抜売上 が **2.0 超** または **0.5 未満**の場合に「桁違いの可能性」を**警告表示**。ただし**保存はブロックしない**（税抜と税込は独立入力のため厳密一致は検証しない）。税抜売上が 0／未入力の場合は判定しない |
| エラー表示 | フィールド直下にインライン表示。桁違い警告は注意喚起スタイル（保存可能なことを明示） |
| 保存タイミング | 明示的な保存ボタン（自動保存なし、誤操作防止） |
| パフォーマンス | 初期表示3秒以内、保存処理2秒以内 |

> **【v1.1 注記】** 旧 v1.0 の「合計整合性（税抜＋税＝税込等）」チェックは**廃止**。総売上（税込）は税抜から自動算出しない独立入力値となったため、厳密な等式チェックは行わず、入力ミス検出を目的とした桁違い警告のみとする（データモデル設計書 v1.7 §8.1.1・要件定義書 v2.4 §4.5.6 と整合）。

#### ⑦部門別売上（参考）セクション（v1.2追加）

**【v1.2：新規】** 部門別（朝食・昼食・夕食・デリバリー等）の売上を把握する**参考機能**。**経営計算（損益・予算比・客単価）には一切反映しない**。データモデル設計書 v1.8 §4.16/§4.17・要件定義書 v2.5 §4.7 と整合。

- 配置：単一ページ縦スクロールの**②売上セクション直下**に独立セクションとして組み込む（タブ化しない）。
- 見出し直下に注記：**「参考データ。経営計算（損益・予算比）には反映されません」**（誤認防止）。
- 行構成：当該店舗の**有効な部門**（部門マスタの `display_order` 順）を一覧表示し、各部門に**税込売上の金額入力欄**を配置。UIは仕入セクションの「一括チェックリスト方式」を踏襲。
- 金額入力欄は**3桁区切りのカンマ表示**（既存の金額入力コンポーネントを再利用）。
- **任意入力**：空欄＝その日その部門の入力なし。**部門別が未入力でも②売上の保存はブロックしない**（保存は独立経路）。
- 部門マスタ未登録の店舗：入力欄を出さず「部門未設定」表示＋部門マスタ画面（#16）への導線のみ。
- 通貨は**店舗ローカル通貨（税込）**。JPY換算は表示しない。
- 整合性：部門別合計と①〜②の1日合計の一致は**検証しない**（警告なし）。

| 設計上の重要点 | 内容 |
|---|---|
| 経営データとの分離 | `daily_department_sales`（専用テーブル）に保存。`daily_sales` とは別の保存アクションで、経営売上の保存・計算に影響しない |
| 保存の独立性 | ②売上と⑦部門別売上は別々に保存可能。⑦が空でも①②の保存は成立 |
| 権限 | 入力は②売上と同一（売上を入力できる人は⑦も入力可） |

## 10. フェーズ5b（個別画面のartifact作成）の進行順

優先順位の高い順に1画面ずつartifact化する。

| 優先順位 | 画面 | 理由 |
|---|---|---|
| 1 | 日次入力 | MVPの中核、UXが最重要 |
| 2 | ログイン | 入口画面、シンプルだが先に固めると他画面のレイアウト判断が早まる |
| 3 | ダッシュボード | ログイン後の最初の画面、ロール別の表示確認 |
| 4 | 売上目標管理 | カレンダーUIの操作性検証 |
| 5 | 仕入先マスタ | 30社以上の登録UXが鍵 |
| 6 | 店舗マスタ | 店舗別パラメータの操作性 |
| 7 | データ閲覧 | フィルタ・テーブル設計 |
| 8 | ユーザー管理 | 管理者専用、頻度低 |
| 9以下 | その他のマスタ画面・設定画面 | 共通パターン化可能 |
